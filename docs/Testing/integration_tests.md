# Curator Integration Test Suite — Design Notes

## Overview

The integration test suite runs all four repository classes against a real
PostgreSQL database (`test_curator` on steward), owned and managed by the
`floater` Postgres role (the Supernumerary). Tests are marked
`@pytest.mark.integration` and skipped by default.

```bash
# Unit tests only (default, no DB required)
pytest

# Integration tests only
pytest -m integration -v

# Both tiers
pytest -m "integration or not integration" -v
```

---

## Infrastructure

| Component | Detail |
|---|---|
| Test database | `test_curator` on steward (`100.64.0.10`) |
| Postgres role | `floater` — `CREATEDB`, no access to `projects` |
| Credentials | `tests/.env.test` (gitignored — password in Proton vault) |
| Schema source | `~/projects/steward/data/projects/schema.sql` |
| Seed source | `~/projects/steward/data/projects/seed.sql` |

The session fixture creates `test_curator` if it doesn't exist, then
bootstraps it by running `schema.sql` and `seed.sql` via `psql`. Because
`schema.sql` is a `pg_dump` output containing psql meta-commands
(`\restrict` etc.), it must be executed via the `psql` client rather than
through psycopg directly. `PGPASSWORD` is passed explicitly to the subprocess
to avoid `.pgpass` lookup issues in the subprocess environment.

`schema.sql` has a full `DROP ... IF EXISTS CASCADE` preamble, so re-running
it is a safe reset — no need to drop and recreate `test_curator` between runs.

---

## Test Isolation

Each test runs inside a psycopg transaction that is rolled back after the
test completes, whether it passes or fails. No teardown scripts are needed
and the database is always clean at the start of each test.

```python
async with conn.transaction():
    yield conn
    raise psycopg.Rollback()   # always rolls back
```

The `lookup` fixture uses a separate `autocommit=True` connection so lookup
table reads (e.g. `task_status`, `priority`) are not affected by the
per-test transaction and do not appear as `idle in transaction` sessions
on steward.

## Known Issues Encountered and Resolved

### 1. `pg_dump` output not executable by psycopg

**Symptom:** `psycopg.errors.SyntaxError: syntax error at or near "\"`

**Cause:** `schema.sql` was generated by `pg_dump` and contains psql
meta-commands (`\restrict`, `\unrestrict`) that are only understood by the
psql client, not by psycopg.

**Fix:** Execute schema and seed via `subprocess.run(["psql", ...])` rather
than `conn.execute()`.

---

### 2. pytest-asyncio event loop mismatch deadlock

**Symptom:** Test suite hangs indefinitely with no output. `py-spy` showed
the process idling in `select()` inside `_async_fixture_wrapper`. On steward,
`pg_stat_activity` showed connections in `idle in transaction / ClientRead`
with `blocked_by = {}` — Postgres had sent replies that Python never received.

**Cause:** Function-scoped async fixtures (`db_conn`, `lookup`, `project_id`)
were opening psycopg connections on a different event loop than the
session-scoped `test_db` fixture. A socket opened on loop A cannot be
awaited on loop B — the `await` blocks forever.

**Fix (partial):** Added `loop_scope="session"` explicitly to every
`@pytest_asyncio.fixture` decorator in `conftest.py`, and added
`asyncio_default_fixture_loop_scope = "session"` to `pyproject.toml`.

**Fix (complete):** Removed `project_id` as an async fixture entirely.
pytest-asyncio 1.x has known reliability issues with function-scoped async
fixtures that depend on session-scoped async fixtures, even with explicit
`loop_scope`. Replaced the `project_id` fixture in all three test files
with a plain `_make_project()` async helper called inline per test. This
eliminated the async fixture dependency chain and resolved the deadlock.

---

### 3. psycopg returning bytes instead of str for text columns

**Symptom:** `assert b'slug-test' == 'slug-test'` — text column values
returned as `bytes` objects.

**Cause:** `_FakeAsyncDBConnection` wraps a psycopg `AsyncConnection` and
creates cursors manually. The connection-level `row_factory=dict_row` was
not being inherited by cursors created via `self._conn.cursor()`.

**Fix:** Pass `row_factory=psycopg.rows.dict_row` explicitly when creating
cursors in `fetch_all` and `fetch_one`. For `fetch_scalar`, use
`next(iter(row.values()))` rather than `next(iter(row))` to return the
value rather than the column name key.
