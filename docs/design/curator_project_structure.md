# Curator — Project Structure

*Annotated directory tree showing where each layer lives.*
*Last updated: April 2026*

## How to Read This Doc

This document answers "where does this go?" for every kind of file in the Curator repo. The tree is intentionally pruned — `__pycache__/`, `.git/`, and the venv are omitted. The `.doc-gen/` directory is also omitted (it's tooling scaffolding, not project code). Doc-gen artifacts appear separately at the end.

│   └── integration/            ← integration tests (psycopg direct, floater role)

│       ├── test_tags_repo.py

├── docker-compose.yml          ← container config (written, never run yet)
├── Dockerfile                  ← container image definition (written, never run yet)
├── .env.example                ← template for tests/.env.test (not committed)

**Notes:**
- `static/` is at repo root, not inside the package. FastAPI mounts it separately via `StaticFiles`.
- `tests/` is at repo root, outside the package, with two subdirectories for the two test tiers. They are run independently: `pytest` (unit) and `pytest -m integration` (integration).
- `go.sh` is machine-specific (contains local paths) and is gitignored. It will be generated by `curator/setup.py` when that is written.
- The venv lives at `venv/` (or wherever the machine puts it). It is gitignored and not part of the project structure.

### Layer responsibilities

Each layer has one job and does not reach into the layer above it.

**`curator/data/`** holds shipped YAML configuration. `curator.yaml` is the app config file. `views.yaml` and `queries.yaml` are the YAML-driven config files read by viewkit. No Python logic lives here. If a user wants to override a view or add a query, they create a file at `~/.config/curator/views.yaml` — the package defaults are never modified.

**`curator/db/`** is the repository layer. Repository classes take an `AsyncDBConnection` (from dbkit) and execute SQL against it. They know nothing about HTTP, FastAPI, or templates. SQL comes from `queries.yaml` via `viewkit.QueryLoader` (fully wired in `projects.py`; remaining repositories are still being backfilled from inline SQL).

**`curator/web/`** is the HTTP layer. Route handlers receive HTTP requests, call repository methods, and return template responses. They know nothing about SQL. `deps.py` is the dependency injection wiring: `get_db` opens and closes a database connection per request; `get_config` returns a `ConfigManager`; `get_query_loader` returns a `QueryLoader` instance. `app.py` assembles the application: creates the FastAPI instance, registers all routers, and mounts `static/`.

**`curator/templates/`** are Jinja2 templates. Templates receive context dicts from routes — they never call Python functions or access the database. `ViewDef` objects from viewkit are passed in as context variables so templates can iterate `view.columns` and `view.fields` without hardcoded column names.

**Route registration order in `app.py` matters.** `/projects/board` must be registered before `/projects/{slug}` or FastAPI treats `board` as a slug. This is correct in code but does not yet have a regression test.

## External Configuration (Outside the Repo)

These files live on the host machine, not in the repository. They are never committed.

```
~/.config/dev-utils/
│   └── config.yaml             ← dbkit reads this: host, port, dbname, user
│                                 Curator does not read or write this file.
│
~/.config/curator/
│   ├── curator.yaml            ← optional user overrides for app config
│   └── views.yaml              ← optional user overrides for view definitions
│
~/.pgpass                       ← PostgreSQL password file; never in code or config
                                  Format: host:port:database:user:password
```

The config hierarchy is: shipped `data/` defaults → user `~/.config/curator/` overrides. Only keys present in the user file are overridden; everything else comes from the package default.

Database credentials flow exclusively through `~/.pgpass`. dbkit reads `~/.config/dev-utils/config.yaml` for connection parameters (host, port, database, user) and delegates password lookup to PostgreSQL's standard `.pgpass` mechanism. Curator never sees a password.

## The Steward Repo (Separate Repository)

Schema and seed SQL live in the Steward repo, not the Curator repo. Curator does not own its own schema — the Steward is the single source of truth for all database structure.

```
steward/                        ← separate repo, lives on the Steward LXC (100.64.0.10)
└── data/
    └── projects/
        ├── schema.sql          ← full PostgreSQL schema for the projects database
        └── seed.sql            ← lookup table seed data (statuses, priorities, types, etc.)
```

Integration tests load these files at setup time. The test fixture fetches them from either a local clone of the Steward repo or via raw GitHub URLs. They are executed via `psql` subprocess (not psycopg) because `pg_dump` output contains psql meta-commands that psycopg cannot execute directly.

## dev-utils Packages (Installed, Not in This Repo)

These packages are installed as editable pip installs (`pip install -e`) from the `dev-utils` repo. They are importable as normal packages but their source lives in a separate repository.

```
~/projects/dev-utils/python/    ← dev-utils repo (separate from Curator)
├── dbkit/
│   └── src/dbkit/
│       ├── connection.py       ← AsyncDBConnection, SyncDBConnection
│       └── config.py           ← ConfigManager (reads ~/.config/dev-utils/config.yaml)
├── viewkit/
│   └── src/viewkit/
│       ├── view_builder.py     ← ViewBuilder: loads views.yaml → ViewDef objects
│       ├── query_loader.py     ← QueryLoader: runtime SQL lookup by entity + name
│       ├── query_builder.py    ← QueryBuilder: parses queries.yaml → QueryDef objects
│       └── query_models.py     ← QueryDef dataclass
├── menukit/
├── mcpkit/
├── fletcher/
└── setupkit/
```

Curator imports from `dbkit` and `viewkit` only. It does not import from menukit, mcpkit, fletcher, or setupkit — those are CLI tools or future integrations.

**Note:** All dev-utils tools currently use a flat layout (source files directly in the package directory, not under `src/`). A refactor to src layout is planned but not yet done. This affects how editable installs are discovered but does not change the import paths.

## "Where Does This Go?" Quick Reference

| What you're adding | Where it lives |
|---|---|
| New route handler | `src/curator/web/routes/{entity}.py` |
| New repository method | `src/curator/db/{entity}.py` |
| New SQL query | `src/curator/data/queries.yaml` under the entity key |
| New template | `src/curator/templates/{entity}/` |
| New form field definition | `src/curator/data/views.yaml` under the entity's `fields:` key |
| New list column definition | `src/curator/data/views.yaml` under the entity's `columns:` key |
| New app config option | `src/curator/data/curator.yaml` (shipped default) |
| New exception class | `src/curator/exceptions.py` |
| New unit test | `tests/unit/test_routes_{entity}.py` |
| New integration test | `tests/integration/test_{entity}_repo.py` |
| New design document | `docs/design/` |
| Change document from a session | `docs/changedocs/{date}/` |
| User config override | `~/.config/curator/` (not in repo) |
| Machine startup script | `go.sh` (gitignored, generated by setup.py) |

**The invariant:** routes never touch SQL; repositories never touch HTTP. If you find SQL in a route file, it belongs in `queries.yaml` and the repository. If you find an HTTP concept in a repository, it belongs in the route.
