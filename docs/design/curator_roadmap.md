# Curator — Roadmap

*Work that is planned, in-progress, or deliberately deferred.*
*Last updated: April 2026*

---

## How to Read This Doc

Items are grouped by domain, not by priority. Each item notes whether it is **actively in progress**, **planned and sequenced**, or **deliberately deferred** (meaning the decision to not do it yet is intentional, not an oversight). See the bottom of this doc for items that are explicitly out of scope for the current development cycle.

---

## Active: In Progress Now

### SQL Externalization — Remaining Route Files

`projects_routes.py` is the reference implementation showing the full `QueryLoader` wiring. The remaining route files still use inline SQL and need to be backfilled:

- `tasks_routes.py`
- `files_routes.py`
- `tags_routes.py`

**Why sequenced this way:** The contacts routes do not exist yet; it makes more sense to wire existing routes before adding new ones.

### contactkit — Gmail Importer MVP

A standalone CLI tool (`contactkit`) for importing contacts from exported CSV/VCF files. The Gmail CSV plugin is implemented with a `BaseImporter` abstract class and `ContactData`/`ImportResult` dataclasses. The tool writes directly into Curator's contacts schema via a dbkit async connection.

**Status:** Gmail importer is implemented. SQL is currently hardcoded in the importer — acceptable for MVP. Remaining work:

- Validate the importer end-to-end against a real export file
- Run import audit logging to `contact_imports`
- Implement remaining format plugins: Proton (VCF), Outlook (CSV), Apple (VCF)

**Intentional sequencing:** The hardcoded SQL will be replaced with `viewkit.QueryLoader` after the domain logic is proven. That refactor also serves as a real-world integration test for QueryLoader outside of Curator. Prove the domain first, then prove viewkit can handle it.

## Planned and Sequenced

### Contacts UI

Schema and import tooling exist; no routes or templates have been written yet.

**Planned scope:**
- Contacts list at `/contacts/` with client-side JS filtering (same pattern as tasks)
- Quick-edit popup on double-click (name and organization only)
- Full edit form at `/contacts/{id}/edit`, gated by role (when auth exists)
- Contacts section in the board panel — tab-style switching alongside tasks

**Dependency:** Role-based auth must exist before the full edit form can be role-gated. Quick-edit and read-only display can be built without auth.

### Redirect Standardization

A comprehensive change document (`REDIRECT_STANDARDIZATION.md`) exists with full BEFORE/AFTER diffs for all CRUD routes. Core decision: `/projects/board` is the single canonical entry point; all form POST redirects should default there. The changes are documented and ready to apply but have not been committed.

### Delete the Standalone Project Detail Page

`GET /projects/{slug}` is superseded by the board panel. It should be removed entirely. A known bug (inline edit redirect) currently lands there — that bug and the deletion should be addressed together.

### Board Panel: Inline Task Edit Redirect Bug

After editing a task from the board panel, the redirect goes to the standalone detail page instead of returning to the board. Fix is blocked on the redirect standardization changes and the detail page deletion — those three things should be done in one pass.

### Route Registration Regression Test

`/projects/board` must be registered before `/{slug}` in the router or FastAPI treats `board` as a slug. This is correct in code but has no test enforcing it. A test should be added before any router refactoring.

### `plugin.yaml` and `curator/setup.py`

Curator does not yet register itself as a Project Crew member. `plugin.yaml` (name, version, description, type) and `curator/setup.py` (imports dbkit setup logic, wires Quartermaster orchestration) have not been written. `go.sh` (machine-specific startup script) will be generated by `setup.py` and gitignored.

### src Layout Refactor for dev-utils Tools

`dbkit`, `viewkit`, `menukit`, and `fletcher` currently use a flat layout. All are planned to be refactored to src layout. This is a dev-utils concern, not a Curator concern, but it will affect editable installs.

## Deliberately Deferred

These items are not planned for the current development cycle. The decision to defer them is intentional.

### Role-Based Authentication and Delete UI

Delete functionality exists in the routes but is not exposed in the UI. Authentication, delete UI, privilege gating, and audit logging are being built together as one cohesive feature. Building any of them in isolation would create partial solutions that need to be torn apart and rebuilt when the others arrive.

**What this means in practice:**
- No user accounts, sessions, or login UI yet
- Delete buttons are not rendered in templates
- The contacts full-edit form cannot be role-gated until this exists
- See `login_design.md` for the design decisions already made (card-based DB auth, session cookies, `.pgpass` lifecycle, brute force protection research needed)

### Containerization

A `Dockerfile` and `docker-compose.yml` exist but have never been run. Containerizing Curator is the next major infrastructure step after the current development cycle stabilizes. The design assumes future migration from bare-metal uvicorn to nginx + containers, and the auth/session design (`login_design.md`) explicitly accounts for container-readiness.

### OTF (On-the-Fly) Query Tool — viewkit.onthefly Integration into Curator

The OTF tool itself is being built as a viewkit CLI tool (design doc: `onthefly_query_tool_design.md`). Integration into Curator as an admin/debug route or board panel query inspector is a possible future addition but is not planned for this cycle. If called from an async Curator route it would need to run in a thread executor.

### Snippets System

Project-scoped searchable reference entries (title, body, free-text category) displayed in the board panel as a third tab alongside Tasks and Contacts. Logged as a future Curator task. Not being designed or implemented yet.

### Subproject Inline Creation

Currently, creating a subproject requires creating it separately and then setting its parent — there is no inline creation flow matching the task creation pattern on the board. Deferred until the board panel UI is more stable.

### setupkit Integration

Curator will eventually expose a `plugin.yaml` and be managed by setupkit for version checking and coordinated installs across machines. Deferred until setupkit's own `init` flow is proven on a fresh clone (planned test: wcyjv10s2).

## Out of Scope (This Project)

- Multi-database support in a single Curator instance (one PostgreSQL database only)
- Simultaneous multi-database sessions (login design explicitly defers this)
- ORM or query builder abstraction (deliberate: SQL is stored as literals for Adminer-debuggability)
- API endpoints (Curator is a server-rendered HTML application; no JSON API is planned)
